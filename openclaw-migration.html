1. Contexto y motivación

Close Energy opera tres agentes de IA en producción (Quoting, Follow-up, Service) orquestados actualmente sobre Kapsos, una plataforma propietaria de agentes que provee: ejecución de pasos, gestión de herramientas, memoria de sesión y triggers de canal (WhatsApp Business API, email, webhooks CRM).

OpenClaw es un framework de orquestación de agentes open-source que expone los mismos primitivos (tool-calling, memory, channel routing) pero bajo un modelo de despliegue autoalojado, con control total del estado, costos predecibles y sin vendor lock-in.

⚠ Consideración crítica
Kapsos gestiona actualmente el estado de sesión de todos los leads en vuelo. La migración en caliente requiere exportar y reanudar ese estado en OpenClaw sin cortar conversaciones activas. Ver sección 7.

Por qué migrar

Kapsos — Limitaciones
  • Pricing por ejecución escala mal con volumen de leads
  • Vendor lock-in en modelo de memoria y tool-registry
  • Logs de conversación cerrados — difícil debuggear
  • Latencia de herramienta no optimizable
  • Sin acceso a traza de razonamiento del modelo
OpenClaw — Ventajas
  • Self-hosted: costo fijo en infra propia / Fly.io / Railway
  • API de herramientas compatible con OpenAI function-calling
  • Memoria persistente en Postgres o Redis — control total
  • Trazas de razonamiento exportables a cualquier observabilidad
  • Soporte nativo de streaming y multi-step tool-call loops

3. OpenClaw — Conceptos clave

OpenClaw es un framework Python async-first para agentes LLM multi-step. Se instala como paquete y se ejecuta en cualquier servidor Python (FastAPI, Starlette) o como worker independiente.

📦 Instalación
pip install openclaw anthropic — OpenClaw usa el Anthropic SDK directamente para tool-calling.

4.4 Scheduler de follow-ups

core/scheduler.py
# KAPSOS (reemplazar):
# kapsos.schedule(agent=FollowUpAgent, lead_id=x, delay_hours=24)

# OPENCLAW (objetivo) — Celery Beat:
from celery import Celery
from openclaw.scheduler import ClawScheduler

app = Celery("close_energy", broker=settings.REDIS_URL)
scheduler = ClawScheduler(celery_app=app)

# Equivalente 1:1 al trigger de Kapsos
async def schedule_followup(lead_id: str, delay_hours: int):
    await scheduler.enqueue_at(
        agent=FollowUpAgent,
        payload={"lead_id": lead_id},
        run_after_hours=delay_hours,
        idempotency_key=f"followup-{lead_id}-{delay_hours}h",
    )

# Secuencia de follow-up: 24h, 48h, 7 días (igual que Kapsos)
for delay in [24, 48, 168]:
    await schedule_followup(lead_id, delay)

El Quoting Agent recibe solicitudes de cotización por WhatsApp, email o web form. Realiza calificación, dimensionamiento técnico, consulta de precios y entrega propuesta personalizada.

agents/quoting.py (versión OpenClaw)
from openclaw import OpenClawAgent
from openclaw.decorators import claw_tool
from tools.solar_sizing import calculate_system_size, estimate_roi
from tools.pricing import lookup_panel_price
from core.outbox import outbox

SYSTEM_PROMPT = """
Eres el Agente de Cotización de Close Energy. Tu misión es...
[system prompt existente sin cambios]
"""

class QuotingAgent(OpenClawAgent):
    model = "claude-sonnet-4-6"
    system_prompt = SYSTEM_PROMPT
    max_iterations = 12
    temperature = 0.2

    # ── Herramientas (firma idéntica a Kapsos) ──

    @claw_tool
    async def calculate_system_size(
        self,
        monthly_kwh: float,
        location: str,
        roof_type: str = "inclinado"
    ) -> dict:
        """Calcula el tamaño óptimo del sistema solar."""
        return await calculate_system_size(monthly_kwh, location, roof_type)

    @claw_tool
    async def lookup_panel_price(
        self,
        panel_model: str,
        quantity: int
    ) -> dict:
        """Consulta precio actual de paneles en catálogo de proveedores."""
        return await lookup_panel_price(panel_model, quantity)

    @claw_tool
    async def estimate_roi(
        self,
        system_kw: float,
        monthly_kwh: float,
        tariff_mxn: float
    ) -> dict:
        """Calcula ROI, período de recuperación y ahorro mensual estimado."""
        return await estimate_roi(system_kw, monthly_kwh, tariff_mxn)

    @claw_tool
    async def send_quote(
        self,
        lead_id: str,
        proposal_markdown: str,
        channel: str
    ) -> dict:
        """Envía la cotización al cliente por el canal indicado."""
        await outbox.send(
            lead_id=lead_id,
            content=proposal_markdown,
            channel=channel,
        )
        return {"status": "sent", "lead_id": lead_id}

5.2 Follow-up Agent

El Follow-up Agent escanea el pipeline, recupera historial de conversación y envía mensajes personalizados en el canal preferido del lead. Es activado por el scheduler en intervalos de 24h, 48h y 7 días.

Diferencia clave en la activación

⚠ Kapsos activaba este agente internamente por evento
En Kapsos el scheduler era un servicio managed. En OpenClaw, el ClawScheduler usa Celery Beat. El entrypoint worker.py debe correr como proceso separado junto a la API.
agents/followup.py (versión OpenClaw)
from openclaw import OpenClawAgent
from openclaw.decorators import claw_tool
from tools.crm import get_lead, update_lead_stage
from core.outbox import outbox

class FollowUpAgent(OpenClawAgent):
    model = "claude-sonnet-4-6"
    system_prompt = """[system prompt existente sin cambios]"""
    max_iterations = 6

    @claw_tool
    async def get_lead_context(self, lead_id: str) -> dict:
        """Recupera perfil, historial y estado de cotización del lead."""
        lead = await get_lead(lead_id)
        history = await self.memory.get_history(session_id=lead_id)
        return {"lead": lead, "history": history}

    @claw_tool
    async def send_followup(
        self,
        lead_id: str,
        message: str,
        channel: str,
    ) -> dict:
        """Envía el mensaje de follow-up al canal preferido del lead."""
        await outbox.send(lead_id=lead_id, content=message, channel=channel)
        await update_lead_stage(lead_id, event="followup_sent")
        return {"status": "sent"}

    @claw_tool
    async def escalate_to_human(self, lead_id: str, reason: str) -> dict:
        """Notifica al equipo de ventas que este lead necesita atención humana."""
        await outbox.notify_team(lead_id=lead_id, reason=reason)
        return {"escalated": True}

7. Migración de datos y estado

⛔ Riesgo crítico: sesiones en vuelo
Kapsos mantiene el historial de conversación activo de cada lead en su memoria propietaria. Este estado no es accesible por REST API — requiere un proceso de exportación antes del cutover.

7.1 Exportar estado de Kapsos

  1. Solicitar a Kapsos el endpoint de exportación de sesiones activas (doc: kapsos.io/docs/export-sessions). Retorna un JSON por session_id con conversation_history, lead_profile y pending_tools.
  2. Ejecutar el script de exportación durante una ventana de baja actividad (ej. 2–4 AM). Las sesiones activas siguen corriendo en Kapsos hasta el cutover.
  3. Transformar el formato Kapsos → formato ClawMemory. El shape es compatible en el 80% — ver transformador abajo.
  4. Importar a Redis (sesiones activas) y Postgres (historial long-term) usando el script de seed.
scripts/migrate_sessions.py
import json, asyncio
from core.memory import memory

async def migrate_kapsos_sessions(export_file: str):
    sessions = json.loads(open(export_file).read())

    for session in sessions:
        session_id = session["session_id"]

        # Transformar formato Kapsos → ClawMemory
        claw_format = {
            "conversation_history": [
                {"role": m["author"], "content": m["text"]}  # renombrar campo
                for m in session["messages"]
            ],
            "lead_profile": session["context"],
            "agent_state": session.get("agent_state", {}),
        }

        await memory.import_session(session_id, claw_format)
        print(f"✓ {session_id}")

asyncio.run(migrate_kapsos_sessions("kapsos_export.json"))

7.2 Migrar embeddings de RAG

Los embeddings generados por KapsosRAG son vectores de 1536 dimensiones (texto-embedding-3-small o equivalente). Para migrar a pgvector:

  1. Exportar embeddings desde Kapsos (endpoint: GET /rag/export?tenant={id}).
  2. Crear tabla en Postgres: CREATE TABLE embeddings (id uuid, namespace text, vector vector(1536), metadata jsonb).
  3. Insertar con pgvector y crear índice HNSW: CREATE INDEX ON embeddings USING hnsw (vector vector_cosine_ops).
  4. Verificar que retrieve_client_docs retorna los mismos top-k que el KapsosRAG en consultas de prueba.

7.3 Scheduler pendientes

Los follow-ups programados en Kapsos (ej. leads que tienen un trigger a las 48h) deben re-encolarse en Celery Beat antes del cutover:

scripts/migrate_scheduled_followups.py
# Exportar desde Kapsos: GET /scheduler/pending → lista de {lead_id, run_at}
for job in pending_jobs:
    delay_secs = (parse_dt(job["run_at"]) - now()).total_seconds()
    if delay_secs > 0:
        await scheduler.enqueue_at(
            agent=FollowUpAgent,
            payload={"lead_id": job["lead_id"]},
            run_after_seconds=delay_secs,
            idempotency_key=f"migrated-{job['id']}",
        )

8. Estrategia de pruebas

8.1 Tests de contrato por herramienta

Cada @claw_tool debe tener un test de contrato que verifique que el output shape es idéntico al de Kapsos antes de cortar el tráfico.

tests/test_tool_contracts.py
import pytest
from agents.quoting import QuotingAgent

@pytest.mark.asyncio
async def test_calculate_system_size_contract():
    agent = QuotingAgent()
    result = await agent.calculate_system_size(
        monthly_kwh=450.0,
        location="CDMX",
    )
    # Mismo shape que devuelve KapsosAgent en la misma llamada
    assert "system_kw" in result
    assert "panels_count" in result
    assert "inverter_model" in result
    assert isinstance(result["system_kw"], float)

8.2 Shadow mode (tráfico paralelo)

Antes del cutover final, ejecutar OpenClaw en shadow mode: el tráfico de producción llega a Kapsos (respuesta real), pero se duplica a OpenClaw en paralelo para comparar outputs sin afectar al usuario.

api.py — shadow mode
from openclaw import shadow_run

@app.post("/webhook")
async def webhook(payload: dict):
    # Producción: Kapsos sigue respondiendo
    kapsos_response = await kapsos.handle(payload)

    # Shadow: OpenClaw corre en background, solo loguea divergencias
    await shadow_run(
        openclaw_agent=router.resolve(payload),
        payload=payload,
        compare_with=kapsos_response,
        log_divergences=True,
    )

    return kapsos_response  # Kapsos sigue siendo la respuesta real

9. Plan de rollout

  1. Semana 1 — Setup de infra: Provisionar Postgres (pgvector), Redis, Celery workers en Fly.io / Railway. Configurar variables de entorno (ANTHROPIC_API_KEY, DATABASE_URL, REDIS_URL, credenciales de canales).
  2. Semana 2 — Migrar código: Refactor de los tres agentes (cambiar imports, decoradores). Tests de contrato al 100%. Deploy de OpenClaw en staging.
  3. Semana 3 — Shadow mode: Activar shadow mode en producción. Observar divergencias durante 5 días hábiles. Ajustar system prompts o tool outputs si hay regresiones.
  4. Semana 4 — Migrar datos: Exportar sesiones y embeddings de Kapsos. Ejecutar scripts de seed. Verificar integridad con checklist de validación.
  5. Día de cutover: Ventana de 2 AM–4 AM. Apagar shadow mode. Redirigir el webhook de WhatsApp/email a la nueva API de OpenClaw. Monitor activo 30 min post-cutover.
  6. Post-cutover: Mantener Kapsos en modo lectura 7 días (no procesar nuevos mensajes) como fallback. Cancelar suscripción al día 8 si todo nominal.
✓ Rollback
Si en las primeras 4h post-cutover la tasa de errores supera el 1%, revertir el webhook apuntando de vuelta a Kapsos. Kapsos sigue activo en modo lectura — no hay pérdida de estado.
